# System Test Guide: Inventory & Production System

This guide outlines the step-by-step process to test the entire system flow, from raw material intake to final product sales. Follow these phases in order to verify full system integrity.

## Phase 1: Prerequisites & Master Data Setup

Before running operations, ensure the fundamental entities exist.

### 1.1 verify Machines
*   **Action**: Go to Admin Portal -> **Machines**.
*   **Check**: Ensure at least one machine exists with status `active`.
*   **Note**: Note the `id` of an "Extruder" type machine for testing.

### 1.2 Verify Products
*   **Action**: Go to Admin Portal -> **Products**.
*   **Check**: Ensure at least one product exists (e.g., "1L Water Bottle") with status `active`.
*   **Important**: Note the `weight_grams` (e.g., 20g) as this drives raw material usage.

### 1.3 Configure Machine-Product Mapping
*   **Action**: Ensure the selected Machine is mapped to the selected Product.
*   **Database Check**: Table `machine_products`.
*   **Why**: Production cannot be submitted if the machine doesn't "know" it can make that product (specifically needs `ideal_cycle_time_seconds`).

---

## Phase 2: Raw Material Management

**Goal**: Ensure there is enough plastic granule stock to produce items.

### 2.1 Check Initial Stock
*   **API**: `GET /inventory/raw-materials`
*   **UI**: Open Mobile App -> Inventory -> Raw Materials.
*   **Expectation**: List of materials (e.g., "Standard Granules").

### 2.2 Add Stock (If Low)
*   **API**: `POST /inventory/raw-materials/:id/adjust`
*   **Payload**: `{ "adjustment": 500, "reason": "Initial Stocking" }` (Adds 500kg).
*   **Verify**: Call GET again to confirm new `stock_weight_kg`.

---

## Phase 3: Production Lifecycle (The "Daily Truth")

**Goal**: Simulate a Production Manager submitting a shift report.

### 3.1 Submit Production Log
*   **API**: `POST /production`
*   **Role**: Production Manager.
*   **Payload**:
    ```json
    {
      "machine_id": "UUID_FROM_PHASE_1",
      "product_id": "UUID_FROM_PHASE_1",
      "shift_number": 1,
      "start_time": "08:00",
      "end_time": "20:00",
      "total_produced": 5000,
      "damaged_count": 100,
      "actual_cycle_time_seconds": 6,
      "actual_weight_grams": 20.5,
      "user_id": "CURRENT_USER_ID"
    }
    ```
*   **Logic Tested**:
    *   **Net Production**: 4900 good units (5000 - 100).
    *   **Time Validation**: Checks shift times and overlaps.

### 3.2 Verify System Effects
1.  **Inventory Increase**:
    *   Check `stock_balances` table.
    *   Find row with `product_id` and `state = 'semi_finished'`.
    *   **Expect**: Quantity increased by **4900**.
2.  **Raw Material Deduction**:
    *   Calculation: 4900 units * 20g (product weight) = 98,000g = **98kg**.
    *   Check `raw_materials`. Stock should be `Initial - 98`.

---

## Phase 4: Mobile App Operations (Inventory Flow)

**Goal**: Simulate floor workers packing and bundling the goods.

### 4.1 Packing (Semi-Finished -> Packed)
*   **Context**: Converting loose items into packets (e.g., 100 items/packet).
*   **API**: `POST /inventory/pack`
*   **Payload**:
    ```json
    {
      "product_id": "UUID",
      "packets_created": 40
    }
    ```
*   **Logic**:
    *   Items needed: 40 packets * 100 items/packet = 4000 items.
    *   **Effect**:
        *   `semi_finished` stock: -4000.
        *   `packed` stock: +40 (Note: System stores Packets count in 'packed' state).

### 4.2 Bundling (Packed -> Finished)
*   **Context**: Converting packets into master bundles/sacks (e.g., 50 packets/bundle).
*   **API**: `POST /inventory/bundle`
*   **Payload**:
    ```json
    {
      "product_id": "UUID",
      "bundles_created": 1
    }
    ```
*   **Logic**:
    *   Packets needed: 1 bundle * 50 packets/bundle = 50 packets.
    *   **Wait!**: We only created 40 packets in step 4.1.
    *   **Test Failure**: Run this to verify **"Insufficient packed stock"** error.
    *   **Correction**: Pack another 10 packets (Step 4.1), then retry.
    *   **Effect**:
        *   `packed` stock: -50.
        *   `finished` stock: +1 (Bundle).

---

## Phase 5: Sales & Order Fulfillment

**Goal**: Sell the finished goods to a customer.

### 5.1 Create Customer
*   **Action**: Admin Portal -> Customers -> Create New.
*   **Name**: "Test Distributor LLC".

### 5.2 Create Sales Order (Reservation)
*   **API**: `POST /sales-orders`
*   **Payload**:
    ```json
    {
      "customer_id": "CUSTOMER_UUID",
      "items": [
        { "product_id": "UUID", "quantity_bundles": 1 }
      ],
      "user_id": "ADMIN_ID"
    }
    ```
*   **Effect**:
    *   Order Status: `reserved`.
    *   `stock_balances` (`finished`): -1.
    *   `stock_balances` (`reserved`): +1.

### 5.3 Deliver Order
*   **API**: `PATCH /sales-orders/:id/status`
*   **Payload**: `{ "status": "delivered", "user_id": "ADMIN_ID" }`
*   **Effect**:
    *   Order Status: `delivered`.
    *   **Key Check**: `stock_balances` (`reserved`) decreases by 1. The stock is now effectively "gone" from the system.

---

## Phase 6: Analytics & Reviews

**Goal**: Confirm the data loop is closed.

### 6.1 Check Production Reports
*   **Report**: **Cycle Time Loss**.
*   **Verify**: Look for the production log from Phase 3.
*   **Check Stats**:
    *   Did `actual_cycle_time_seconds` (6s) differ from `ideal`?
    *   verify `units_lost_to_cycle` is calculated.

### 6.2 Check Customer Analytics
*   **API**: `GET /customers/:id/analytics`
*   **Verify**:
    *   `total_orders`: 1
    *   `delivered_orders`: 1
    *   `last_purchase_date`: Today.

---
**End of Test Procedure**
